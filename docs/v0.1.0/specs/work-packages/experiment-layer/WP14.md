# **WP14 – ExperimentExecutor**

## **Context**

We are implementing the **Experimentation Framework** on top of the already completed AXIS System A runtime.

At this point, the following layers already exist:

* the complete episode-level runtime and execution pipeline
* structured `EpisodeResult` objects
* passive logging / observability
* `RunExecutor`, `RunConfig`, `RunSummary`, `RunResult` from WP11
* `ExperimentConfig`, canonical experiment types, deterministic run resolution, `ExperimentSummary`, and `ExperimentResult` from WP12
* the repository and persistence layer from WP13, including:

  * deterministic experiment/run artifact layout
  * save/load support for configs, results, metadata, and status
  * experiment and run status models
  * artifact discovery helpers

This means the framework already has:

* a working **run-level execution unit**
* a working **experiment-level configuration and result model**
* a working **filesystem persistence contract**

What is still missing is the orchestration layer that turns one `ExperimentConfig` into a fully executed persisted experiment.

The architecture defines this layer explicitly as the **ExperimentExecutor**, sitting above the `RunExecutor` and below any later CLI interface:

```text
ExperimentExecutor
    └── RunExecutor
            └── EpisodeRunner
```

This layering is a hard requirement. The `ExperimentExecutor` must orchestrate runs, not execute episodes directly, and it must remain above the runtime and below the CLI. 

---

## **Objective**

Implement **WP14 – ExperimentExecutor**.

The goal of this package is to create a correct, explicit, and testable top-level orchestration component that:

1. accepts one validated `ExperimentConfig`
2. resolves it into one or more concrete `RunConfig` objects
3. creates and persists the experiment artifact structure
4. executes runs sequentially through the existing `RunExecutor`
5. persists experiment-level metadata and status
6. builds the final `ExperimentSummary`
7. constructs and persists the final `ExperimentResult`

This package must implement the **happy-path baseline execution** of full experiments.

This is **not yet** the fault-tolerance and resume package.
That comes next.

---

## **Scope**

Implement only the following.

---

### **1. ExperimentExecutor as Top-Level Orchestrator**

Implement a dedicated `ExperimentExecutor` component.

Its responsibility is to orchestrate one complete experiment from:

* one `ExperimentConfig`
* one repository root / repository instance
* one `RunExecutor`

into:

* persisted experiment artifacts
* persisted run artifacts
* final in-memory `ExperimentResult`

The executor must be the only component responsible for:

* coordinating the sequence of runs belonging to one experiment
* maintaining experiment-level lifecycle state during execution
* constructing the final experiment-level result

It must **not**:

* execute episodes directly
* compute run-level policy/transition behavior
* duplicate config expansion logic from WP12
* duplicate repository IO logic from WP13

It is a thin but explicit orchestration layer.

---

### **2. Inputs and Dependencies**

The `ExperimentExecutor` must consume at least:

* an `ExperimentConfig`
* an `ExperimentRepository`
* a `RunExecutor`

It may also consume optional execution options if these are truly execution-scoped and do not drift into CLI concerns.

A reasonable constructor shape could be:

```python
class ExperimentExecutor:
    def __init__(
        self,
        repository: ExperimentRepository,
        run_executor: RunExecutor,
    ) -> None:
        ...
```

And a main entry point such as:

```python
def execute(self, config: ExperimentConfig) -> ExperimentResult:
    ...
```

Equivalent naming is acceptable if it matches the existing code style.

Important:

* do not let the repository create runs automatically
* do not let the CLI logic leak in here
* do not embed hidden global dependencies

---

### **3. Experiment Initialization**

Implement explicit initialization of a new experiment execution.

This includes:

* determining the experiment ID
* creating the experiment directory via the repository
* persisting the input `ExperimentConfig`
* creating and persisting `ExperimentMetadata`
* initializing and persisting `ExperimentStatusRecord` with `CREATED`, then `RUNNING`

Important constraints:

* experiment identity must be explicit and stable
* if the config already carries an explicit experiment identifier or name, respect that
* if IDs are derived inside the executor, the derivation must be deterministic and transparent
* do not silently overwrite an existing experiment unless an explicit future policy allows it, which is not part of this package

The architecture requires explicit lifecycle/status handling and deterministic artifact creation. 

---

### **4. RunConfig Resolution and Run Planning**

Use the already implemented WP12 resolution logic to expand the input `ExperimentConfig` into one or more concrete `RunConfig` objects.

This package must:

* call the existing resolution logic
* preserve the stable order of resolved runs
* assign or finalize stable run identifiers if not already fully resolved
* create an explicit run plan in memory for sequential execution

Important:

* do not reimplement OFAT logic here
* do not mutate the meaning of the resolved runs
* do not introduce ad hoc run filtering or selection yet
* the run plan must reflect the exact resolved experiment definition

The architecture explicitly requires that experiment execution operates on fully resolved run configurations. 

---

### **5. Run Artifact Initialization**

Before each run is executed, initialize its repository artifacts.

For each resolved run, the `ExperimentExecutor` must:

* create the run directory
* persist `RunConfig`
* persist `RunMetadata`
* persist `RunStatusRecord` with `PENDING`, then `RUNNING` when actual run execution begins

This package is responsible for the experiment-level orchestration around runs.
The `RunExecutor` remains responsible for actually producing `RunResult`.

Important:

* use the repository methods from WP13
* do not embed raw path logic here
* keep the lifecycle explicit

---

### **6. Sequential Run Execution**

Execute all runs **sequentially**.

This is a hard baseline architecture rule.

For each run in the resolved run plan:

1. mark run status as `RUNNING`
2. invoke the existing `RunExecutor`
3. collect the returned `RunResult`
4. persist the returned `RunResult`
5. persist the returned `RunSummary`
6. mark run status as `COMPLETED`

Important constraints:

* no parallel execution
* no process pools
* no background scheduling
* no speculative concurrency flags

The architecture deliberately defers parallelization in favor of simplicity, deterministic ordering, and easier debugging. 

---

### **7. Failure Handling in the Happy Path Baseline**

WP14 should implement **explicit failure propagation**, but not full resume or recovery behavior yet.

If one run fails during experiment execution:

* the exception must not be silently swallowed
* the failing run must be marked as `FAILED`
* the experiment status must be updated to `FAILED` or `PARTIAL` according to the chosen architecture-consistent rule
* already completed runs must remain persisted and valid
* the exception may propagate after status updates are written

Important:

* do not implement skip/retry/recovery logic yet
* do not recompute already completed runs automatically
* do not attempt resume behavior inside this package

A reasonable baseline choice is:

* if any run fails during a fresh experiment execution, write:

  * the failed run as `FAILED`
  * the experiment as `FAILED` if no later runs were attempted
  * or `PARTIAL` if some runs completed and then one failed

This should be implemented explicitly and consistently.

The architecture is clear that completed work must remain valid and visible, but detailed resume behavior belongs later. 

---

### **8. ExperimentSummary Construction**

After all runs complete successfully, compute the final `ExperimentSummary` using the already implemented WP12 summary logic.

This must be done from the collected run-level outputs, not from logs.

Requirements:

* use `RunResult` and/or `RunSummary` as the source of truth
* preserve stable run ordering
* include baseline-relative deltas if applicable under the existing summary model
* keep summary computation deterministic

Then persist the resulting `ExperimentSummary` via the repository.

This follows the architectural rule that summaries are derived artifacts built from structured results, not from logs. 

---

### **9. ExperimentResult Construction**

After the experiment finishes successfully, construct the final `ExperimentResult`.

It must contain at minimum:

* the original `ExperimentConfig`
* the ordered collection of run references and/or `RunResult` objects
* the final `ExperimentSummary`

Then persist the resulting `ExperimentResult` if your repository already supports this artifact directly.

If the repository contract currently stores `ExperimentSummary` and the individual run artifacts, but not `ExperimentResult` as a separate file, that is acceptable only if it is fully consistent with the existing WP13 artifact design.

Important:

* no live executor references
* no filesystem handles
* no mutable state objects
* keep it fully serializable

---

### **10. Final Experiment Status Update**

At the end of a successful execution, the `ExperimentExecutor` must update the experiment status to `COMPLETED`.

A clean successful finalization sequence should therefore be:

1. experiment initialized as `CREATED`
2. experiment moved to `RUNNING`
3. runs executed sequentially
4. final summary/result persisted
5. experiment status moved to `COMPLETED`

This lifecycle ordering should be explicit and testable.

---

## **Out of Scope**

Do **not** implement any of the following in WP14:

* resume logic
* restart-from-partial logic
* skipping already completed runs
* retry policies
* CLI commands
* CLI formatting/output
* ad hoc interactive inspection
* parallel run execution
* database-backed experiment storage
* dashboards or analytics UIs
* changes to `RunExecutor`
* changes to `EpisodeRunner`
* changes to runtime simulation logic
* generalized execution scheduling frameworks

Do not let WP14 turn into the resume package.
Do not let it turn into the CLI package.

---

## **Architectural Constraints**

The implementation must follow these rules.

### **1. Strict orchestration layering**

The hierarchy must remain:

```text
ExperimentExecutor
    └── RunExecutor
            └── EpisodeRunner
```

The `ExperimentExecutor` must never execute episodes directly. It operates only through `RunExecutor`. 

---

### **2. No duplication of lower-layer logic**

Do not duplicate:

* config resolution from WP12
* repository path/IO logic from WP13
* run summary computation from WP11/WP12
* runtime execution from WP7/WP11

The `ExperimentExecutor` coordinates. It does not reinvent.

---

### **3. Results remain the source of truth**

The executor must build experiment-level outputs from:

* `RunResult`
* `RunSummary`
* `ExperimentSummary`

Never from:

* console logs
* JSONL logs
* inferred side effects

This is a hard design rule. 

---

### **4. Deterministic run ordering**

Given the same `ExperimentConfig` and the same run-resolution logic, the execution order of runs must be identical.

No hidden sorting changes, no opportunistic reordering, no randomization.

---

### **5. Explicit lifecycle state handling**

All state transitions must be written explicitly through the repository artifacts:

* experiment status
* run status

Do not keep lifecycle state only in memory.

---

### **6. Keep it simple**

Do not introduce:

* schedulers
* orchestration graphs
* workflow engines
* callback registries
* plugin-based executor chains

A plain sequential orchestration loop is the correct baseline here.

---

## **Expected File Structure**

A reasonable extension could be:

```text
src/axis_system_a/
    run.py
    experiment.py
    repository.py
    experiment_executor.py
    __init__.py
```

Or, if you prefer fewer files:

```text
src/axis_system_a/
    run.py
    experiment.py
    repository.py
    execution.py
    __init__.py
```

The important thing is that:

* experiment execution is clearly separated from config/result models
* experiment execution is clearly separated from repository logic
* the executor layer remains easy to inspect and test

Do not fragment the code excessively.

---

## **Testing Requirements**

Also create pytest tests for WP14.

At minimum include the following.

### **Happy-path execution tests**

* `single_run` experiment executes successfully
* `ofat` experiment executes all resolved runs successfully
* experiment artifacts are created as expected
* run artifacts are created as expected
* final `ExperimentSummary` is persisted correctly
* final `ExperimentResult` is returned and, if applicable, persisted correctly

### **Ordering tests**

* runs execute in the resolved deterministic order
* `ExperimentSummary` preserves that same order

### **Status lifecycle tests**

* experiment status transitions through `CREATED -> RUNNING -> COMPLETED`
* run status transitions through `PENDING -> RUNNING -> COMPLETED`
* statuses are actually written to repository artifacts

### **Failure tests**

* if one run raises an exception:

  * that run is marked `FAILED`
  * already completed runs remain valid
  * experiment status becomes `FAILED` or `PARTIAL` consistently
  * exception propagation behavior is explicit and tested

### **Determinism tests**

* same `ExperimentConfig` + same seeds → same run plan and same resulting experiment summary
* no hidden reordering occurs

### **Repository integration tests**

* executor uses repository helpers rather than raw path handling
* persisted artifacts can be reloaded correctly after execution

Use temporary directories and small handcrafted experiment definitions.
