# **WP12 – Experiment Configuration and Result / Summary Consolidation**

## **Context**

We are implementing the **Experimentation Framework** on top of the already completed AXIS System A runtime.

The core runtime up to WP10 is already available and stable, including:

* world, observation, agent state, memory
* hunger drive
* policy and decision pipeline
* transition engine
* episode execution
* structured episode-level results
* passive logging / observability

In addition, **WP11 is already implemented** and introduced the first experimentation-layer component:

* `RunExecutor`
* `RunConfig`
* `RunSummary`
* `RunResult`
* deterministic episode-seed generation inside a run
* multi-episode execution over a shared runtime configuration

This means the framework already has a functioning **run-level execution unit**.

However, the current state is still incomplete with respect to the Experimentation Framework architecture:

* experiment-level configuration is not yet standardized
* the architecture-defined result hierarchy is not yet complete
* the current run summary is intentionally minimal and must now be aligned with the architecture-defined summary schema
* experiment-level result and summary objects do not yet exist

The architecture is explicit that the Experimentation Framework extends the result hierarchy from:

```text
EpisodeResult → RunResult → ExperimentResult
```

and that summaries are deterministic, derived views built from structured results, not from logs.

This package must therefore consolidate and extend the current state without breaking the already implemented run execution behavior.

---

## **Objective**

Implement **WP12 – Experiment Configuration and Result / Summary Consolidation**.

The goal of this package is to establish the **formal data contracts** required for the remainder of the Experimentation Framework.

This includes:

1. **Experiment-level configuration models**
2. **Experiment type modeling**
3. **Run-summary schema alignment with the architecture**
4. **Experiment-level result and summary structures**
5. **Deterministic summary computation rules**
6. **Explicit handling of partial and failed data at the model level**

This package must provide the stable structural foundation required for later implementation of:

* persistence
* experiment orchestration
* resume behavior
* CLI inspection

This package is **not** responsible for executing full experiments yet.
That comes later with `ExperimentExecutor`.

---

## **Scope**

Implement only the following.

---

### **1. ExperimentConfig**

Introduce an explicit `ExperimentConfig` model.

This model defines the top-level input for an experiment.

At minimum it must support:

* experiment identifier or optional explicit name
* experiment type
* baseline runtime configuration
* run generation definition
* optional base seed
* optional metadata fields relevant to orchestration
* optional persistence-related settings only if they are purely declarative and do not require repository logic yet

Important:

* `ExperimentConfig` lives above `RunConfig`
* it does **not** replace `RunConfig`
* it is responsible for defining how one or more run configurations are derived

The architecture defines experiment-level configuration as the source for run expansion and explicitly distinguishes it from run-level resolved configuration.

---

### **2. Canonical Experiment Types**

Implement explicit support for the initial canonical experiment types defined by the architecture:

* `single_run`
* `ofat`

These must be represented explicitly and validated.

#### **single_run**

Represents an experiment that resolves to exactly one run.

#### **ofat**

Represents a One-Factor-at-a-Time experiment built from:

* one complete baseline configuration
* one parameter path
* one set of variation values

Important constraints:

* only one parameter may vary per OFAT definition
* all other parameters remain fixed
* no ad hoc multi-parameter sweep logic
* no grid-search generalization
* no plugin-style experiment-type registry

The architecture is intentionally conservative here to prevent complexity from expanding too early. 

---

### **3. RunConfig Resolution from ExperimentConfig**

Provide deterministic resolution logic that expands `ExperimentConfig` into one or more concrete `RunConfig` instances.

This is one of the most important responsibilities of WP12.

Resolution must produce:

* fully materialized `RunConfig` objects
* immutable run-level configuration snapshots
* deterministic run ordering
* deterministic run-level seed derivation if a base seed is provided

Important:

* this package should resolve configurations
* it must **not** execute them
* it must **not** persist them
* it must **not** depend on the future `ExperimentExecutor`

This logic should likely live as:

* pure resolver functions
* or a small `ExperimentConfigResolver`

Keep it simple and explicit.

The architecture explicitly requires that runs own fully resolved configuration snapshots and that no unresolved variation rules remain at run level.

---

### **4. Parameter Addressing for OFAT**

Implement stable path-based parameter addressing used by OFAT variation resolution.

Examples are explicitly given in the architecture, such as:

* `agent.energy.initial`
* `policy.beta`
* `environment.resource.regen_rate`

Requirements:

* paths must map directly to the configuration structure
* invalid paths must fail explicitly
* path resolution must be deterministic
* no reflective “best effort” guessing
* no dynamic schema generation
* no runtime mutation of the config shape

This package should define the minimal utility needed to:

* resolve a path
* override a value at that path in a copied config object
* validate that the target path exists

The architecture explicitly prefers simple, explicit schemas over dynamic config machinery. 

---

### **5. Consolidate RunSummary to Architecture Schema**

`RunSummary` already exists from WP11, but it is currently minimal.

WP12 must now align it with the architecture-defined summary schema.

The minimal required `RunSummary` schema must include:

* `num_episodes`
* `mean_steps`
* `std_steps`
* `mean_final_energy`
* `std_final_energy`
* `death_rate`
* `mean_consumption_count`
* `std_consumption_count`

This replaces the earlier intentionally small sanity-check summary as the formal run-level summary contract. The architecture defines this schema explicitly.

Important:

* do not remove already useful fields unless truly redundant
* additive extension is preferred over breaking reshaping
* field naming should now stabilize

---

### **6. Deterministic Run Summary Computation**

Implement deterministic aggregation from `EpisodeResult[]` to `RunSummary`.

Computation rules:

* only use structured `EpisodeResult` data
* never depend on logs
* computation must be reproducible
* computation must be independent of execution order
* missing or invalid episode outputs must be handled explicitly

This package should likely revise or extend the WP11 `compute_run_summary()` behavior rather than replacing it wholesale.

Important:

* no histogram support yet
* no fancy descriptive statistics
* no visualization-facing transforms
* just the architecture-required baseline metrics

The architecture is clear that summaries are derived views, deterministic, and recomputable at any time.

---

### **7. Introduce ExperimentSummary**

Implement a new `ExperimentSummary` structure.

This object aggregates across multiple runs.

At minimum it must support:

* per-run entry containing:

  * `run_id`
  * key variation descriptor
  * `RunSummary`
* optional stable ordering of runs
* optional baseline-relative comparison fields if a baseline is defined

Baseline comparison, if present, should support at least:

* delta in mean steps
* delta in death rate
* delta in energy metrics

This is explicitly defined in the architecture and should be reflected now at the model level, even if the actual executor and persistence of experiment results comes later.

---

### **8. Introduce ExperimentResult**

Implement a new `ExperimentResult` structure.

It must contain:

* reference to `ExperimentConfig`
* collection of run references and/or `RunResult`s
* experiment metadata fields
* `ExperimentSummary`

Important:

* keep it fully serializable
* no live references to executors, runtime objects, or filesystem handles
* this is a passive result object only

The architecture defines `ExperimentResult` as the top-level comparative result artifact of the experimentation layer.

---

### **9. Explicit Handling of Partial and Failed Data**

WP12 must formalize model-level handling of incomplete or failed execution data.

At the **run level**:

* failed episodes must be recordable
* summaries must clearly indicate:

  * number of valid episodes
  * number of failed episodes
* failed episodes must be excluded or flagged explicitly, not silently ignored

At the **experiment level**:

* failed runs must remain visible
* they must not invalidate other runs
* experiment summaries must be able to represent partial experiment state cleanly

This does **not** yet implement resume logic.
It only defines the data model and summary behavior needed for it later.

The architecture is explicit here and this matters a lot, because later persistence and resume depend on these contracts being honest.

---

### **10. Serialization Readiness and Schema Stability**

All new and revised structures introduced in WP12 must be:

* fully serializable
* JSON-compatible
* stable in naming and structure
* suitable for later persistence without redesign

Provide minimal helpers such as:

* `to_dict()`
* optional `from_dict()` if the current project style already supports this cleanly

Important:

* avoid speculative schema flexibility
* avoid untyped dict soup
* avoid runtime-dependent fields
* do not attach persistence paths yet

The architecture explicitly requires minimal schema stability and warns against frequent breaking changes, because later CLI, persistence, and visualization will depend on these fields.

---

## **Out of Scope**

Do **not** implement any of the following in WP12:

* repository / filesystem persistence
* directory creation
* status-file writing
* `ExperimentExecutor`
* resume logic
* CLI commands
* parallel execution
* database storage
* visualization logic
* experiment dashboards
* dynamic plugin-based experiment strategies
* generalized grid search or random search
* mutation of runtime execution semantics
* changes to `EpisodeRunner`
* changes to core runtime policy / transition / observation logic

Do not let WP12 turn into a persistence or executor package.

---

## **Architectural Constraints**

The implementation must follow these rules.

### **1. Respect the WP11 reality**

`RunExecutor`, `RunConfig`, `RunSummary`, and `RunResult` already exist.

WP12 must extend and align them, not pretend they do not exist.

---

### **2. No back-dependency into runtime**

Nothing in WP12 may require changes to:

* policy
* transition
* sensor
* world
* agent state
* `EpisodeRunner`

The experimentation layer stays above the runtime.

---

### **3. Results remain the source of truth**

Summaries must always be derived from results, not logs.

No summary computation may depend on:

* console output
* JSONL logs
* external analysis state

This is a hard design rule.

---

### **4. Determinism is mandatory**

Given identical:

* `ExperimentConfig`
* baseline config
* parameter variation definitions
* seeds

the package must resolve the same run configs and produce the same summaries.

No hidden randomness is allowed in resolution or aggregation.

---

### **5. Keep the schemas explicit**

Do not introduce:

* config DSLs
* schema generators
* reflection-heavy config systems
* generic sweep frameworks
* experiment-type plugin registries

The architecture explicitly biases toward controlled simplicity.

---

## **Expected File Structure**

A reasonable extension could be:

```text
src/axis_system_a/
    run.py
    experimentation_config.py
    experimentation_results.py
    experimentation_resolution.py
    __init__.py
```

Or, if you prefer fewer files:

```text
src/axis_system_a/
    run.py
    experiment.py
    __init__.py
```

The important point is not the exact filenames, but the separation of concerns:

* run execution stays in the existing run-oriented module
* experiment config / resolution is separated from execution
* experiment result structures are explicit

Do not fragment the package excessively.

---

## **Testing Requirements**

Also create pytest tests for WP12.

At minimum include the following.

### **Configuration tests**

* valid `ExperimentConfig` for `single_run`
* valid `ExperimentConfig` for `ofat`
* invalid experiment type fails
* invalid parameter path fails
* invalid OFAT variation definition fails
* unresolved baseline config shape fails explicitly if inconsistent

### **Resolution tests**

* `single_run` resolves to exactly one `RunConfig`
* `ofat` resolves to one run per variation value
* resolved runs preserve stable ordering
* same config + same seeds → identical resolved `RunConfig`s
* baseline config remains unchanged after resolution

### **RunSummary tests**

* all required architecture fields are present
* mean/std/death-rate/consumption metrics are computed correctly
* summary ignores logs completely
* same episode result set in different order produces the same summary, if order independence is the chosen rule for aggregation input handling

### **ExperimentSummary tests**

* valid construction from multiple runs
* run IDs and variation descriptors are preserved
* baseline comparison fields computed correctly when baseline exists
* failed runs remain visible and do not collapse the summary

### **Serialization tests**

* all new models are JSON-serializable
* `to_dict()` output is stable and complete
* no live runtime references leak into result objects

---

## **Implementation Style**

* Python 3.11
* explicit typing
* simple dataclasses or pydantic-style models only if already consistent with the project
* pure functions for resolution and aggregation where practical
* no hidden global state
* no unnecessary abstraction layers

---

## **Expected Deliverable**

1. experiment-level config models
2. canonical experiment type support (`single_run`, `ofat`)
3. deterministic run-resolution logic
4. architecture-aligned `RunSummary`
5. new `ExperimentSummary`
6. new `ExperimentResult`
7. tests covering resolution, summary computation, determinism, and serialization

---

## **Important Final Constraint**

WP12 must feel like:

> a **schema and aggregation package**, not an execution package.

Execution beyond the already existing run layer belongs later.
