# **WP13 – Persistence Layer and Repository Model**

## **Context**

We are implementing the **Experimentation Framework** on top of the already completed AXIS System A runtime.

At this point, the following layers already exist:

* the complete episode-level runtime and execution pipeline
* structured `EpisodeResult` objects
* passive logging / observability
* `RunExecutor`, `RunConfig`, `RunSummary`, `RunResult` from WP11
* `ExperimentConfig`, canonical experiment types, run-resolution logic, `ExperimentSummary`, and `ExperimentResult` from WP12

This means the framework already has:

* execution at the episode and run level
* experiment-level configuration and expansion
* a complete result hierarchy

However, these objects are not yet backed by a **formal persistence layer**.

The Experimentation Framework architecture is explicit that experiment execution must be based on **file-system-based persistence**, with:

* deterministic directory layout
* explicit artifact files
* immutable result artifacts
* metadata and status files
* incremental writes
* support for later resume and inspection workflows 

WP13 must now implement this persistence layer as a dedicated repository subsystem.

This package must remain strictly scoped to:

* artifact layout
* filesystem path resolution
* serialization / deserialization
* load / store operations

It must **not** absorb execution logic, CLI logic, or resume control.

---

## **Objective**

Implement **WP13 – Persistence Layer and Repository Model**.

The goal of this package is to create a correct, explicit, and testable persistence subsystem for the Experimentation Framework.

This includes:

1. a deterministic repository root model
2. experiment and run directory resolution
3. file naming and artifact-path conventions
4. serialization and deserialization of persisted artifacts
5. immutable write semantics for result artifacts
6. repository access helpers for later executors and CLI commands

This package must provide the persistence foundation required for later implementation of:

* `ExperimentExecutor`
* fault tolerance and resume
* CLI-based inspection
* post-hoc analysis over stored artifacts

The architecture is clear that persistence is not an optional detail. It is part of reproducibility, auditability, and controlled recovery. 

---

## **Scope**

Implement only the following.

---

### **1. Repository Root Model**

Introduce a repository abstraction for the experimentation artifact root.

At minimum, this package must support a configured repository root such as:

```text
experiments/
```

or an explicitly configured alternative root path.

The repository root is the entry point for all experiment and run artifact storage.

Important requirements:

* path handling must be explicit and platform-safe
* no hidden current-working-directory assumptions
* no hardcoded magic paths inside executors later
* root existence may be created explicitly if missing, but only in a controlled way

The architecture defines the repository root as the filesystem anchor for all experiment artifacts. 

---

### **2. Deterministic Experiment Directory Layout**

Implement deterministic path resolution for one experiment directory.

The architecture already defines the expected experiment layout, including files such as:

* `experiment_config.json`
* `experiment_metadata.json`
* `experiment_status.json`
* `experiment_summary.json`

and nested run directories under:

```text
runs/<run_id>/
```

WP13 must realize this artifact layout faithfully. 

At minimum, provide helpers to resolve:

* experiment root directory
* path to each experiment-level artifact file
* path to the `runs/` directory under one experiment

Important:

* directory naming must be stable
* no executor logic here
* no inference from partial files yet
* this package only defines and accesses the layout

---

### **3. Deterministic Run Directory Layout**

Implement deterministic path resolution for one run inside one experiment.

The architecture defines run-level artifacts such as:

* `run_config.json`
* `run_metadata.json`
* `run_status.json`
* `run_summary.json`
* `run_result.json`

and nested episode result files under:

```text
episodes/
    episode_0001.json
    ...
```

with optional logs under:

```text
logs/
    run.log.jsonl
```

WP13 must provide path helpers for this structure. 

Important requirements:

* stable run directory structure
* deterministic episode filename formatting
* support for listing existing episode artifact files
* logs path resolution may be included as a path helper only, not as a logging implementation

---

### **4. Artifact File Contracts**

Implement persistence support for the artifact types already defined or implied by the architecture.

At minimum, the persistence layer must support reading and writing the following categories:

#### **Experiment-level artifacts**

* `ExperimentConfig`
* experiment metadata object
* experiment status object
* `ExperimentSummary`
* `ExperimentResult` if persisted separately by later execution logic

#### **Run-level artifacts**

* `RunConfig`
* run metadata object
* run status object
* `RunSummary`
* `RunResult`

#### **Episode-level artifacts**

* `EpisodeResult`

The persistence layer should not invent new schemas.
It should use the already existing data contracts from WP8, WP11, and WP12.

---

### **5. Serialization and Deserialization**

Provide explicit serialization helpers for all persisted artifact types.

At minimum this package should support:

* `save_*()` functions or repository methods
* `load_*()` functions or repository methods
* JSON serialization
* JSON-compatible deserialization into the corresponding model objects

Important constraints:

* do not persist Python object graphs directly
* do not pickle
* use explicit structured serialization only
* fail explicitly on malformed persisted content
* no “best effort” reconstruction from invalid files

This aligns with the architecture’s insistence on explicit, inspectable, immutable artifacts. 

---

### **6. Metadata and Status Artifact Support**

The architecture distinguishes between:

* configuration artifacts
* result artifacts
* metadata artifacts
* status artifacts

WP13 must support persistence for metadata and status objects, even if some of their lifecycle transitions are only exercised later by executors and resume logic.

At minimum, this package should define lightweight, explicit models or typed dictionaries for:

#### **Experiment metadata**

Potential fields may include:

* `experiment_id`
* creation timestamp
* experiment type
* optional label / description
* baseline config reference or summary information if appropriate

#### **Run metadata**

Potential fields may include:

* `run_id`
* `experiment_id`
* variation descriptor
* creation timestamp
* seed information or seed strategy reference if appropriate

#### **Experiment status**

At minimum must support the statuses already defined in the architecture:

* `CREATED`
* `RUNNING`
* `COMPLETED`
* `FAILED`
* `PARTIAL`

#### **Run status**

At minimum must support:

* `PENDING`
* `RUNNING`
* `COMPLETED`
* `FAILED`

These models should remain simple and passive.
Do not implement status transition policy here. That comes later. 

---

### **7. Immutable Write Semantics for Result Artifacts**

The architecture explicitly treats configurations and results as **immutable artifacts**. WP13 must respect that.

This means:

* persisted config snapshots must not be silently overwritten
* persisted result artifacts must not be silently mutated after the fact
* writes should fail explicitly if overwrite is attempted where immutability is required

A practical design is acceptable, for example:

* config/result writes default to `overwrite=False`
* metadata/status writes may support controlled overwrite where lifecycle updates require it later

Important:

* do not overcomplicate this into a transaction system
* just make the immutability boundary explicit and enforceable

This is important because later resume behavior depends on artifact trustworthiness. 

---

### **8. Listing and Discovery Helpers**

Provide repository-level helper functions for artifact discovery.

At minimum, this package should support:

* list experiment IDs under repository root
* list run IDs within one experiment
* list episode artifact files within one run
* check whether specific artifacts exist

These are not yet resume operations.
They are repository inspection utilities that later packages will use for:

* executor orchestration
* CLI commands
* fault tolerance and resume

Keep these helpers explicit and deterministic.

---

### **9. Reconstruction Support**

The persistence layer must support reconstructing persisted structured artifacts back into the in-memory model types.

At minimum, support reconstruction of:

* `ExperimentConfig`
* `RunConfig`
* `RunSummary`
* `RunResult`
* `ExperimentSummary`
* `ExperimentResult`
* `EpisodeResult`

Important constraints:

* reconstruction must be strict
* invalid or incomplete JSON content must fail explicitly
* do not silently fill missing required fields with guessed defaults

The architecture explicitly prefers correctness and inspectability over permissive convenience. 

---

### **10. Repository Facade or Equivalent**

Implement the persistence layer behind one clear access abstraction.

A good solution would be something like:

```python
class ExperimentRepository:
    ...
```

with methods such as:

* `create_experiment_dir(...)`
* `save_experiment_config(...)`
* `load_experiment_config(...)`
* `save_run_config(...)`
* `load_run_config(...)`
* `save_episode_result(...)`
* `load_episode_result(...)`
* `list_experiments()`
* `list_runs(experiment_id)`
* `list_episode_files(experiment_id, run_id)`

Equivalent explicit functions are also acceptable.

What matters is:

* one clearly scoped persistence layer
* no scattered file IO across later executors
* no duplication of path logic

---

## **Out of Scope**

Do **not** implement any of the following in WP13:

* `ExperimentExecutor`
* run or experiment execution control
* resume logic
* skip / retry policy
* status transition policy
* CLI commands
* logging output writing beyond path conventions
* database storage
* cloud storage backends
* dashboards or analysis UIs
* metrics computation
* config expansion logic
* mutation of runtime behavior

Do not let WP13 turn into a resume package or an executor package.

---

## **Architectural Constraints**

The implementation must follow these rules.

### **1. Persistence is passive**

The repository layer stores and loads artifacts.
It does not decide:

* when a run should execute
* when a run should be resumed
* whether a result is valid semantically
* how statuses advance

It only handles artifacts and path conventions.

---

### **2. No logic duplication**

Do not duplicate:

* config expansion from WP12
* run summary computation from WP11/WP12
* experiment summary computation from WP12
* logging semantics from WP10

The repository persists artifacts. It does not recreate the framework.

---

### **3. Deterministic artifact layout**

Given identical experiment IDs and run IDs, the repository must resolve identical paths.

No hidden randomness, timestamps, or automatic suffixing should alter artifact locations unless explicitly requested by the caller.

---

### **4. Strict separation between mutable and immutable artifacts**

Treat:

* config snapshots
* summaries
* results

as immutable by default.

Treat:

* metadata
* status

as lifecycle-managed artifacts that may need controlled updates later.

Make this distinction explicit in the API.

---

### **5. No back-dependency into runtime**

The repository layer must not depend on:

* policy
* transition
* observation
* agent internals
* world internals

It works with structured contracts only.

---

## **Expected File Structure**

A reasonable implementation could extend the package like this:

```text
src/axis_system_a/
    run.py
    experiment.py
    repository.py
    repository_models.py
    __init__.py
```

Or, if you want fewer files:

```text
src/axis_system_a/
    run.py
    experiment.py
    repository.py
    __init__.py
```

The important thing is to keep:

* path logic
* IO logic
* metadata/status models

inside one clear repository-oriented area.

Do not fragment excessively.

## **Testing Requirements**

Also create pytest tests for WP13.

At minimum include the following.

### **Repository root and path tests**

* repository resolves experiment directory correctly
* repository resolves run directory correctly
* repository resolves episode file path with stable naming
* repository listing helpers work on small handcrafted temp directories

### **Write / read roundtrip tests**

* `ExperimentConfig` can be saved and loaded roundtrip
* `RunConfig` can be saved and loaded roundtrip
* `EpisodeResult` can be saved and loaded roundtrip
* `RunSummary` can be saved and loaded roundtrip
* `RunResult` can be saved and loaded roundtrip
* `ExperimentSummary` can be saved and loaded roundtrip
* `ExperimentResult` can be saved and loaded roundtrip

### **Immutability tests**

* saving a config/result artifact twice without overwrite fails explicitly
* metadata/status overwrite behavior is controlled and explicit

### **Discovery tests**

* list experiments returns expected IDs
* list runs returns expected run IDs
* list episode artifact files returns expected sorted filenames

### **Failure tests**

* malformed JSON fails explicitly on load
* missing file fails explicitly
* invalid artifact content fails explicitly
* wrong directory assumptions are not silently tolerated

Use temporary directories and small handcrafted artifacts only.

---

## **Implementation Style**

* Python 3.11
* use `pathlib`
* explicit typing
* explicit JSON read/write helpers
* no hidden magic
* no general-purpose persistence framework
* no ORM-like abstraction
* no pickling

Keep the repository simple, deterministic, and boring. That is a compliment here.

---

## **Expected Deliverable**

1. repository abstraction and root handling
2. deterministic experiment/run/episode path resolution
3. artifact save/load support for configs, summaries, results, metadata, and status
4. immutable artifact write semantics where required
5. discovery/listing helpers
6. strict reconstruction support
7. pytest coverage for path logic, roundtrips, immutability, discovery, and failure cases

---

## **Important Final Constraint**

WP13 must feel like:

> a **filesystem contract package**, not a workflow package.
