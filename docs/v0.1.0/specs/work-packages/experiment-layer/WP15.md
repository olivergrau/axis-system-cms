# **WP15 – Fault Tolerance and Resume**

## **Context**

We are implementing the **Experimentation Framework** on top of the completed AXIS System A runtime.

At this point, the following layers already exist:

* the complete runtime and episode execution pipeline
* structured `EpisodeResult` objects
* passive logging / observability
* `RunExecutor`, `RunConfig`, `RunSummary`, and `RunResult` from WP11
* `ExperimentConfig`, run-resolution logic, `ExperimentSummary`, and `ExperimentResult` from WP12
* the repository and artifact model from WP13
* the happy-path `ExperimentExecutor` from WP14

This means the framework can already:

* resolve experiments into runs
* execute those runs sequentially
* persist experiment and run artifacts
* persist episode results incrementally
* mark run and experiment lifecycle states
* fail explicitly if something goes wrong

However, the framework still lacks the recovery behavior explicitly required by the architecture:

* interrupted experiments must be restartable
* already completed work must be preserved
* completed runs must not be recomputed unintentionally
* partial execution must remain inspectable
* resume must operate from persisted artifacts and statuses, not from guessed in-memory assumptions 

The architecture is also very clear about the baseline resume granularity:

> the recommended baseline resume boundary is the **run**.
> Completed runs are skipped, incomplete or missing runs are executed.
> Episode-level resume is optional and not required for the baseline. 

WP15 must now implement this recovery behavior as a dedicated layer.

---

## **Objective**

Implement **WP15 – Fault Tolerance and Resume**.

The goal of this package is to extend the Experimentation Framework with robust, explicit, and testable resume behavior that:

1. detects persisted experiment state from the repository
2. distinguishes completed, incomplete, and failed runs
3. skips already completed runs
4. re-executes missing or incomplete runs
5. preserves already completed artifacts
6. reconstructs final experiment state correctly after resume
7. does all of this without changing runtime semantics or duplicating execution logic

This package must make experiment execution **resumable and idempotent at the run boundary**.

---

## **Scope**

Implement only the following.

---

### **1. Resume Entry Point for Experiment Execution**

Introduce an explicit resume-capable execution entry point.

A reasonable design would be one of these:

* extend `ExperimentExecutor` with a dedicated `resume(experiment_id: str)` method
* or add a dedicated `ResumeExecutor`
* or add a clearly scoped `resume_experiment(...)` helper using the existing executor and repository

The important part is not the exact name, but the semantics:

* resume must start from an existing persisted experiment
* it must use repository artifacts as source of truth
* it must not rely on transient in-memory state from a previous run

Important:

* do not hide resume logic behind the normal fresh `execute(...)` path unless that behavior is explicit and well separated
* the API should make it obvious whether a caller is starting fresh or resuming existing work

The architecture treats resume as a distinct capability with explicit status handling and persisted state inspection. 

---

### **2. Experiment State Reconstruction from Repository**

Implement the ability to reconstruct the current persisted execution state of an experiment.

At minimum, resume logic must load and inspect:

* `ExperimentConfig`
* `ExperimentMetadata`
* `ExperimentStatusRecord`
* run directories under the experiment
* `RunConfig`
* `RunStatusRecord`
* any existing `RunSummary`
* any existing `RunResult`

Important requirements:

* reconstruction must be strict
* malformed artifacts must fail explicitly
* do not silently “repair” broken persisted structures
* do not infer experiment meaning from missing files beyond clearly defined repository checks

The repository from WP13 already provides load and discovery helpers. WP15 must use those, not bypass them. 

---

### **3. Run-Level Resume Boundary**

Implement run-level resume as the baseline recovery unit.

This is a hard constraint.

That means:

* if a run is already `COMPLETED` and its required artifacts are present, it must be skipped
* if a run is `FAILED`, `RUNNING`, `PENDING`, or its required completion artifacts are missing or incomplete, it is eligible for re-execution
* partial episode artifacts inside a non-completed run do **not** trigger episode-level resume
* the entire run is re-executed from scratch when necessary

Important:

* no episode-level replay or continuation
* no attempt to “continue from episode 7 of 10”
* no hybrid resume logic
* keep the baseline recovery rule simple and explicit

This is directly aligned with the architecture’s stated primary resume boundary. 

---

### **4. Completion Detection Rules**

Define and implement explicit rules for deciding whether a run counts as completed and therefore must be skipped.

A run should be considered resumably complete only if:

* `RunStatusRecord.status == COMPLETED`
* `run_config.json` exists and loads correctly
* `run_summary.json` exists and loads correctly
* `run_result.json` exists and loads correctly

Depending on the existing repository and executor behavior, you may also require expected episode artifacts to be present if that is already a stable contract.

Important constraints:

* do not rely on status file alone if completion artifacts are missing
* do not rely on artifact presence alone if status says otherwise
* the rule must be explicit and deterministic

The architecture emphasizes that resume logic must detect completed vs incomplete units and avoid recomputing already finished work unintentionally. 

---

### **5. Incomplete and Failed Run Handling**

Implement explicit handling for non-completed runs.

At baseline, the following run states should be resumable by re-executing the run:

* `PENDING`
* `RUNNING`
* `FAILED`

Also resumable are structurally incomplete runs, for example:

* missing `run_result.json`
* missing `run_summary.json`
* malformed persisted result artifact
* partially written episode files without run completion

The behavior for those cases should be:

* mark the run as resumable / incomplete
* re-execute the whole run
* overwrite only the mutable lifecycle/status artifacts as needed
* preserve immutable prior artifacts when they are valid and belong to already completed runs
* for incomplete runs being re-executed, controlled cleanup or overwrite policy must be explicit

Important:

* do not try to salvage partial episode sequences for baseline resume
* do not attempt file-by-file healing logic
* do not leave the run in ambiguous mixed old/new state after re-execution

---

### **6. Experiment-Level Resume Planning**

Implement explicit planning of which runs to skip and which runs to execute during resume.

The resume flow should be conceptually:

1. load the persisted experiment
2. reconstruct the original run plan from `ExperimentConfig`
3. compare planned runs with persisted run artifacts
4. classify runs into:

   * already completed
   * needs execution / re-execution
5. execute only the missing or incomplete runs, in deterministic run order
6. rebuild final experiment-level summary/result from:

   * previously completed runs
   * newly completed runs

Important constraints:

* ordering must stay stable
* completed runs must preserve their position in the plan
* no ad hoc run discovery order
* no opportunistic reordering

This follows directly from the architecture’s requirement that resume be incremental and deterministic. 

---

### **7. Experiment Status Semantics During Resume**

Implement explicit experiment-level status behavior during resumed execution.

A sensible baseline behavior is:

* if an experiment is resumed, set status to `RUNNING` during active resume execution
* if all required runs complete successfully after resume, set experiment status to `COMPLETED`
* if some runs remain incomplete due to a new failure, set experiment status to `PARTIAL` or `FAILED` consistently with the already established status semantics

Important:

* status updates must be persisted
* do not keep final truth only in memory
* do not leave the experiment in stale `RUNNING` after a handled failure path

The architecture treats lifecycle status as part of execution control and inspection, not as decorative metadata. 

---

### **8. Idempotent Resume Behavior**

Implement idempotent resume at the experiment level.

This means:

* resuming an already fully completed experiment must not recompute any runs
* it should either:

  * return the reconstructed completed `ExperimentResult`
  * or perform a no-op verification pass and then return the persisted result state

Likewise:

* repeated resume calls on a partially completed experiment with no new failures should converge to the same final completed artifact set
* resume must not duplicate work
* resume must not create duplicate run directories or altered run identities

This idempotency requirement is essential for operational reliability. The architecture is explicit that completed work must never be recomputed unintentionally. 

---

### **9. Controlled Re-Execution of Incomplete Runs**

When a run is classified as incomplete and needs re-execution, implement a controlled policy for how the repository is reused.

At minimum, the policy must be explicit for:

* `run_status.json`
* `run_summary.json`
* `run_result.json`
* episode artifacts under `episodes/`

A good baseline approach is:

* treat incomplete runs as re-executable
* before re-execution, clear or replace incomplete mutable run artifacts in a controlled way
* then execute the run fresh
* persist the new run result set deterministically

Important:

* do not silently append new episode artifacts to stale incomplete data unless that behavior is explicitly correct
* do not mix stale partial run outputs with fresh outputs
* keep the baseline policy simple and safe

This is one of those places where being conservative is better than being clever.

---

### **10. Final Experiment Reconstruction After Resume**

After all remaining runs have been executed successfully, reconstruct and persist the final experiment-level outputs exactly as in normal execution:

* `ExperimentSummary`
* `ExperimentResult`
* final `ExperimentStatusRecord` with `COMPLETED`

Important:

* the summary must include both:

  * previously completed runs
  * newly completed runs
* the final order must match the original resolved run plan
* baseline-relative deltas must still compute correctly for OFAT experiments
* no distinction should remain between “freshly completed” and “completed before resume” at the final summary level

This preserves the architectural principle that final summaries are deterministic derived artifacts from structured results. 

---

## **Out of Scope**

Do **not** implement any of the following in WP15:

* episode-level resume
* partial replay of a run from the middle
* retry counts / exponential backoff
* automatic corruption repair
* CLI commands
* human-facing inspect/report formatting
* parallel resume execution
* distributed recovery
* database-backed checkpointing
* speculative artifact healing
* changes to runtime semantics
* changes to `EpisodeRunner`
* changes to `RunExecutor` beyond what is strictly needed to support clean re-execution

Do not let WP15 become a workflow engine.

---

## **Architectural Constraints**

The implementation must follow these rules.

### **1. Persisted artifacts are the source of truth**

Resume must operate from:

* repository artifacts
* status records
* persisted configs/results/summaries

Not from:

* cached executor state
* logs
* ad hoc assumptions about previous execution

---

### **2. Run boundary is the baseline resume unit**

Do not implement episode-level continuation.
If a run is incomplete, re-execute the whole run. 

---

### **3. No duplication of execution logic**

Do not duplicate:

* run execution logic from WP11
* config resolution from WP12
* repository IO logic from WP13
* experiment summary computation from WP12
* happy-path orchestration from WP14

Resume extends orchestration control. It does not replace the framework.

---

### **4. Completed work must be preserved**

A completed run with valid completion artifacts must not be recomputed during resume. This is a hard requirement. 

---

### **5. Explicit failure over silent repair**

If persisted artifacts are malformed in a way that prevents safe classification or reconstruction, fail explicitly.

Do not silently guess or heal.

---

### **6. Keep the baseline conservative**

Do not try to optimize for maximum salvage of partial work.
Optimize for:

* correctness
* determinism
* inspectability
* stable behavior

That is the right baseline here.

---

```text

    run.py
    experiment.py
    repository.py
    experiment_executor.py
    resume.py
    __init__.py
```

Or, if you prefer to extend the existing experiment executor module:

```text
src/axis_system_a/
    run.py
    experiment.py
    repository.py
    experiment_executor.py
    __init__.py
```

with resume support clearly separated inside that module.

The important thing is conceptual separation:

* normal fresh execution remains understandable
* resume logic remains explicit and testable
* repository logic stays in the repository layer

Do not fragment the package excessively.

## **Testing Requirements**

Also create pytest tests for WP15.

At minimum include the following.

### **Completed-experiment resume tests**

* resuming a fully completed experiment does not re-execute any runs
* completed experiment returns reconstructed final state cleanly
* no duplicate artifacts are created

### **Partial-experiment resume tests**

* if one or more runs are already completed, those runs are skipped
* only incomplete runs are executed
* final experiment summary includes both skipped-completed and newly executed runs

### **Run classification tests**

* `COMPLETED` + valid artifacts → skipped
* `FAILED` → re-executed
* `RUNNING` from interrupted prior execution → re-executed
* `PENDING` with missing result artifacts → re-executed
* malformed or incomplete completion artifacts fail classification explicitly or trigger safe re-execution according to the chosen rule, but behavior must be fully tested and documented

### **Status lifecycle tests**

* experiment status changes correctly during resume
* re-executed runs get proper status transitions
* final status is `COMPLETED` after successful recovery

### **Idempotency tests**

* repeated resume on already completed experiment is a no-op
* repeated resume after a partial state converges correctly
* no completed run is executed twice unintentionally

### **Artifact integrity tests**

* completed run artifacts remain unchanged after resume
* incomplete run artifacts are replaced or cleaned according to the explicit policy
* final persisted experiment artifacts are loadable and consistent

### **Failure propagation tests**

* if a resumed run fails again, experiment ends in `FAILED` or `PARTIAL` consistently
* already completed runs remain valid and preserved

Use temporary directories and small handcrafted experiment setups.

This package should read like a careful operational recovery layer, not like a framework within a framework.
