# **WP10 – Logging and Observability Layer**

## **Objective**

Introduce a **passive logging and observability layer** that:

* Consumes the structured runtime outputs defined in **WP8 (Result and Trace Structures)**
* Produces **human-readable and structured logs**
* Does **not influence system behavior or decision-making**
* Remains **lightweight, deterministic, and fully optional**

The logging system must act purely as an **output projection layer**, not as a source of truth.

---

## **Scope**

### **In Scope**

* Logging of:

  * `StepResult`
  * `EpisodeResult`
* Structured rendering of:

  * `DecisionTrace`
  * `TransitionTrace`
* Console output (human-readable)
* File output (JSONL structured logs)
* Configurable logging behavior

### **Out of Scope**

* No metrics system (e.g. Prometheus)
* No distributed tracing
* No real-time dashboards
* No coupling with policy or transition logic
* No mutation of runtime objects

---

## **Design Principles**

### **1. Results Are the Source of Truth**

All logging must be derived from:

* `StepResult`
* `EpisodeResult`
* `RunResult`

No logging logic may reconstruct or reinterpret internal state.

---

### **2. Strict Separation of Concerns**

| Layer         | Responsibility               |
| ------------- | ---------------------------- |
| Core System   | Produces results and traces  |
| Logging Layer | Renders and persists results |

---

### **3. Passive and Non-Intrusive**

* Logging must not:

  * affect execution order
  * modify state
  * introduce randomness
* Logging failures must **never crash execution**

---

### **4. Deterministic Output**

Given identical inputs and seeds:

* Logging output must be reproducible
* Ordering of logs must match execution order

---

### **5. Minimal Dependency Philosophy**

* Use **Python standard `logging` module**
* No heavy external frameworks
* Small AXIS-specific abstraction layer on top

---

## **Architecture Overview**

### **1. Core Components**

#### **AxisLogger (Facade)**

Central entry point:

```python
class AxisLogger:
    def log_step(self, step_result: StepResult): ...
    def log_episode(self, episode_result: EpisodeResult): ...
    def log_run(self, run_result: RunResult): ...
```

Responsibilities:

* Delegates formatting
* Routes to configured outputs
* Applies filtering rules

---

#### **Formatters**

##### **StepFormatter**

* Converts `StepResult` into:

  * human-readable string
  * structured dict

##### **EpisodeFormatter**

* Aggregates episode-level data

##### **RunFormatter**

* Aggregates multi-episode summaries

---

#### **Trace Renderers**

Optional detailed formatting:

* `DecisionTraceRenderer`
* `TransitionTraceRenderer`

These must support:

* compact mode
* verbose mode

---

#### **Output Handlers**

Built on top of Python logging:

* **ConsoleHandler**
* **JsonFileHandler (JSONL)**

---

## **Logging Modes**

### **1. Human-Readable Mode (Console)**

Example:

```text
[STEP 12] pos=(3,4) energy=0.72 action=MOVE_RIGHT
  decision: consume=0.12 move=0.88 selected=MOVE_RIGHT
```

Goals:

* readable during debugging
* compact
* no excessive nesting

---

### **2. Structured Mode (JSONL)**

Each line = one event:

```json
{
  "type": "step",
  "step": 12,
  "position": [3,4],
  "energy": 0.72,
  "action": "MOVE_RIGHT",
  "decision_trace": {...},
  "transition_trace": {...}
}
```

Goals:

* machine-readable
* reproducible
* usable for offline analysis

---

## **Configuration**

### **LoggingConfig**

```python
@dataclass
class LoggingConfig:
    enabled: bool = True

    log_level: str = "INFO"

    console_enabled: bool = True
    jsonl_enabled: bool = False

    jsonl_path: Optional[str] = None

    include_decision_trace: bool = True
    include_transition_trace: bool = True

    verbosity: str = "compact"  # or "verbose"
```

---

## **Integration Points**

### **1. Episode Execution Loop (WP7)**

Logging hooks:

```python
for step in episode:
    step_result = execute_step(...)
    logger.log_step(step_result)

logger.log_episode(episode_result)
```

---

### **2. Run-Level Execution**

```python
for episode in run:
    ...

logger.log_run(run_result)
```

---

### **3. No Deep Integration**

Logging must **not** be embedded inside:

* policy
* transition function
* drives
* sensor model

---

## **Error Handling**

* Logging failures must be:

  * caught
  * reported once
* System execution must continue

Example:

```python
try:
    logger.log_step(step_result)
except Exception:
    pass
```

---

## **Performance Considerations**

* Logging must be:

  * lightweight
  * optionally disabled
* JSON serialization must be efficient
* Avoid deep copies of large structures

---

## **Test Requirements**

### **1. Determinism Tests**

* Same input → identical logs

---

### **2. Formatting Tests**

* Validate structure of JSON output
* Validate presence/absence of traces

---

### **3. Integration Tests**

* Ensure logging does not alter results
* Ensure execution continues on logging failure

---

### **4. Config Tests**

* Toggle console/json modes
* Toggle verbosity
* Toggle trace inclusion

---

## **Extension Boundaries**

The logging system must allow future extensions:

* additional handlers (e.g. remote logging)
* metrics extraction layer (future WP)
* visualization tools
* replay systems

But:

> No extension may violate the passive nature of logging.

---

## **Acceptance Criteria**

WP10 is complete when:

1. All result types can be logged:

   * Step
   * Episode
   * Run

2. Both output modes exist:

   * Console (human-readable)
   * JSONL (structured)

3. Logging is:

   * deterministic
   * optional
   * non-intrusive

4. Integration with WP7 works cleanly

5. Tests confirm:

   * no behavioral side effects
   * correct formatting
   * config works

---

## **Final Note**

This WP is intentionally conservative.

The goal is not observability sophistication.
The goal is **clarity and traceability of a mechanistic system**.

If WP10 is done right, you gain:

* debuggability
* reproducibility
* analyzable behavior

Without polluting the core system.
