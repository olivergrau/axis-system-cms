# **WP11 – Run Orchestration Core**

## **1. Objective**

The objective of WP11 is to introduce a **Run-level orchestration layer** that:

* Executes **multiple episodes under a shared configuration**
* Aggregates episode-level outputs into a **RunResult**
* Provides a **clean control boundary** between:

  * Execution (Episode loop, already implemented in WP7)
  * Experimentation (statistics, persistence, CLI, later WPs)

This is **not** yet about persistence, CLI, or advanced statistics.

This is about **structural correctness and control flow**.

---

## **2. Scope**

### **In Scope**

* Run abstraction
* Run configuration binding
* Multi-episode execution loop
* EpisodeResult aggregation
* Basic RunResult structure
* Deterministic execution (seed handling)

### **Out of Scope**

* Persistence (WP13)
* Advanced statistics (WP12)
* CLI integration (WP15)
* Parallel execution (deliberately excluded for now)
* Resume / fault tolerance (WP16)

---

## **3. Conceptual Model**

We introduce a new hierarchy level:

```
Run
 ├── Configuration (shared)
 ├── Episodes (N executions)
 │    └── EpisodeResult
 └── RunResult (aggregated view)
```

Key idea:

> A **Run is a controlled experiment** with repeated stochastic realizations.

Each episode is:

* Independent in execution
* Identically configured (except seed)

---

## **4. Core Components**

### **4.1 RunConfig**

Defines parameters for a run.

```python
@dataclass
class RunConfig:
    num_episodes: int
    base_seed: Optional[int]  # None = non-deterministic
    max_steps_per_episode: int

    # Optional extensions (keep minimal for now)
    run_id: Optional[str] = None
    description: Optional[str] = None
```

---

### **4.2 RunContext**

Internal runtime object.

```python
@dataclass
class RunContext:
    run_config: RunConfig
    episode_seeds: List[int]
```

---

### **4.3 RunResult**

Minimal structure (do NOT over-engineer here).

```python
@dataclass
class RunResult:
    run_id: str
    num_episodes: int

    episode_results: List[EpisodeResult]

    # Basic aggregated values (only minimal!)
    mean_total_reward: float
    mean_steps: float

    # Optional metadata
    seeds: List[int]
```

Important constraint:

* No deep statistical analysis yet
* Only **simple aggregates required for sanity checks**

---

### **4.4 RunExecutor**

Main orchestration class.

```python
class RunExecutor:

    def __init__(self, episode_executor: EpisodeExecutor):
        self.episode_executor = episode_executor

    def run(self, config: RunConfig) -> RunResult:
        ...
```

---

## **5. Execution Flow**

### **5.1 Seed Generation**

If `base_seed` is provided:

```
episode_seed_i = base_seed + i
```

If not:

* Use random seeds (but still store them!)

Skeptical note:
Do not rely on global RNG state. Seeds must be explicit.

---

### **5.2 Main Loop**

Pseudo-flow:

```
initialize RunContext

for i in range(num_episodes):
    seed = episode_seeds[i]

    episode_result = episode_executor.run(
        max_steps=config.max_steps_per_episode,
        seed=seed
    )

    collect episode_result

aggregate results

return RunResult
```

---

### **5.3 Aggregation (Minimal)**

Only compute:

* mean_total_reward
* mean_steps

Explicitly avoid:

* variance
* percentiles
* histograms

Reason:

> Keep WP11 focused. Statistics belong to WP12.

---

## **6. Interfaces to Existing System**

### **Dependency**

* Must reuse existing:

  * `EpisodeExecutor`
  * `EpisodeResult`

### **Assumption**

`EpisodeExecutor.run(...)` supports:

```python
run(max_steps: int, seed: Optional[int]) -> EpisodeResult
```

If not:

→ This must be adapted inside WP11 (small wrapper allowed)

---

## **7. Determinism Requirements**

If `base_seed` is set:

* Full run must be reproducible
* Same config → identical RunResult (within floating point tolerance)

This is **non-negotiable**, otherwise later analysis is meaningless.

---

## **8. Error Handling**

Minimal but strict:

* If an episode fails → fail the entire run
* Do NOT silently skip episodes

Reason:

> Silent failure corrupts statistical validity

Retry logic comes later (WP16)
