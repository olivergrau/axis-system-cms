# **WP16 – CLI Interface and End-to-End Integration Tests**

## **Context**

We are implementing the **Experimentation Framework** on top of the completed AXIS System A runtime.

At this point, the following layers already exist:

* the complete runtime and episode execution pipeline
* structured `EpisodeResult` objects
* passive logging / observability
* `RunExecutor`, `RunConfig`, `RunSummary`, `RunResult`
* `ExperimentConfig`, experiment types, deterministic run resolution, `ExperimentSummary`, `ExperimentResult`
* the persistence layer and repository model
* `ExperimentExecutor` for fresh experiment execution
* fault tolerance and resume support at the run boundary

This means the framework already supports the core operational behaviors required by the architecture:

* executing new experiments
* persisting artifacts incrementally
* reconstructing experiment state from disk
* resuming incomplete experiments without duplicating completed runs
* producing deterministic summaries and results from persisted artifacts 

What is still missing is the first real **user-facing entry point** for using the whole experimentation system.

The architecture defines this as a **minimal, scriptable CLI** that sits above the executors and repository, and provides direct operational access to experiments and runs. It must expose real persisted data, not hidden presentation abstractions, and it must remain stateless between invocations. 

---

## **Objective**

Implement **WP16 – CLI Interface and End-to-End Integration Tests**.

The goal of this package is to create a correct, explicit, and testable command-line interface that:

1. executes new experiments from config files
2. resumes interrupted or partial experiments
3. lists experiments in the repository
4. shows experiment-level persisted data
5. lists runs belonging to an experiment
6. shows run-level persisted data
7. does all of this through the existing executors and repository layer
8. proves the full framework works end-to-end through integration and e2e tests

This package must expose the framework operationally without changing its underlying semantics.

---

## **Scope**

Implement only the following.

---

### **1. CLI Entry Point**

Implement a top-level CLI entry point following the architecture’s command structure:

```text
axis <entity> <action> [arguments]
```

with:

* `<entity> ∈ { experiments, runs }`
* `<action>` as the operation to perform

This command structure is explicitly defined in the architecture and must be followed directly. 

A baseline implementation should provide a single executable entry point, for example:

* `axis`
* or equivalent packaging entry point if already consistent with the project

Important:

* no interactive prompts
* no REPL-like mode
* no hidden shell state
* argument-driven only

The CLI must be non-interactive by default. 

---

### **2. Core Commands**

Implement the baseline core commands defined by the architecture.

#### **A. List Experiments**

```bash
axis experiments list
```

Purpose:

* list all experiments in the repository root

Output should expose persisted experiment-level information such as:

* experiment ID
* status
* number of runs
* number of completed runs, if derivable cheaply and explicitly
* creation timestamp if present in metadata

This command is explicitly required by the architecture. 

---

#### **B. Run Experiment**

```bash
axis experiments run <path_to_experiment_config>
```

Purpose:

* execute a new experiment from configuration

Behavior:

* load and validate the config file
* create the experiment through `ExperimentExecutor`
* persist all artifacts through existing framework layers
* fail explicitly if the experiment already exists under baseline semantics

The architecture explicitly states that `run` should fail if the experiment already exists, unless future policy changes allow otherwise.

---

#### **C. Resume Experiment**

```bash
axis experiments resume <experiment_id>
```

Purpose:

* resume an interrupted or partial experiment

Behavior:

* load existing persisted experiment state
* invoke existing resume logic
* execute only missing or incomplete runs
* preserve completed runs

The architecture explicitly requires that CLI resume respects resume semantics and does not override them.

---

#### **D. Show Experiment**

```bash
axis experiments show <experiment_id>
```

Purpose:

* inspect a single experiment from persisted artifacts

Output should include persisted information such as:

* experiment ID
* experiment status
* experiment metadata
* effective configuration
* summary
* run references or run IDs
* relevant artifact references or paths where appropriate

This command must read from repository artifacts, not from in-memory executor state.

---

#### **E. List Runs**

```bash
axis runs list --experiment <experiment_id>
```

Purpose:

* list all runs belonging to an experiment

Output should expose persisted run-level information, such as:

* run ID
* status
* variation descriptor if available
* summary presence
* artifact presence if appropriate

This command is explicitly part of the required CLI test surface. 

---

#### **F. Show Run**

```bash
axis runs show <run_id>
```

Purpose:

* inspect one persisted run

According to the architecture, the output should include:

* resolved configuration
* run status
* run summary
* episode count
* references to stored artifacts 

Because run IDs are scoped by experiment in the repository layout, you may implement this in one of two acceptable ways:

* require globally unique run IDs if the current system already guarantees that
* or require an experiment context if needed for safe lookup

But whichever choice is made must be explicit and deterministic. Do not guess.

---

### **3. Minimal Flags**

Implement only the essential baseline flags defined or strongly suggested by the architecture.

At minimum support flags such as:

* `--root <path_to_experiment_repository>`
* `--output json`
* `--experiment <experiment_id>` where relevant

The architecture explicitly says the initial CLI should support only essential flags and that advanced filtering/querying/formatting is out of scope. 

Do **not** implement:

* rich filtering
* ad hoc search syntax
* table formatting frameworks
* colorized interactive output
* arbitrary export backends

Keep the baseline CLI minimal.

---

### **4. Config File Loading**

Implement loading of experiment configuration files for:

```bash
axis experiments run <path_to_experiment_config>
```

Requirements:

* support the config format already used by the framework, at minimum JSON, and YAML only if it is already part of the accepted architecture and current project conventions
* validate before execution
* fail explicitly on malformed or invalid config
* keep config-loading logic in the CLI layer limited to file parsing plus handing off to `ExperimentConfig` validation

Do not turn the CLI into a second configuration engine.

---

### **5. Output Philosophy**

The CLI must expose **real system data**, not invented presentation abstractions.

The architecture is explicit here:

* output reflects persisted artifacts directly
* no hidden aggregation beyond defined summaries
* prefer structured outputs where appropriate
* future human-readable modes are allowed, but the baseline must remain transparent and scriptable

A good baseline approach is:

* default to a clear human-readable text mode
* support `--output json` for machine/script use

Important:

* JSON output must map directly to persisted or reconstructed model objects
* do not inject ad hoc derived fields without architectural justification
* do not format away important state distinctions like `FAILED` vs `PARTIAL`

---

### **6. Thin Wrapper Constraint**

The CLI must act only as a thin wrapper over the execution and repository layers.

It may:

* parse arguments
* load config files
* instantiate repository/executors
* invoke:

  * `ExperimentExecutor.execute(...)`
  * `ExperimentExecutor.resume(...)`
  * repository read/list methods
* render outputs

It must **not**:

* reimplement experiment orchestration
* reimplement resume logic
* reimplement run summary computation
* bypass the repository with manual path handling
* add business logic that diverges from the framework

This is a hard architectural constraint.

---

### **7. Stateless CLI Behavior**

The architecture requires that the CLI remain stateless between invocations.

That means:

* no hidden caches
* no session state
* no dependency on previous command execution in memory
* every command must reconstruct what it needs from:

  * arguments
  * config files
  * persisted repository artifacts

This matters especially for:

* `experiments show`
* `experiments resume`
* `runs show`
* `runs list`

The CLI must operate on persisted artifacts only.

---

### **8. Error Handling**

The CLI must fail **explicitly and clearly**.

The architecture explicitly gives examples such as:

* unknown experiment ID → error
* invalid config → error before execution
* missing artifacts → explicit failure message
* no silent fallbacks
* no implicit default behavior 

Therefore the CLI must:

* return non-zero exit codes on failure
* surface clear error messages
* avoid swallowing exceptions silently
* avoid ambiguous “best effort” behavior

This applies particularly to:

* unknown experiment IDs
* unknown run IDs
* missing repository root
* malformed config files
* artifact reconstruction failures

---

### **9. Respect Resume and Idempotency Semantics**

The CLI must not override the framework’s fault tolerance rules.

Specifically:

* `axis experiments resume <id>` must continue incomplete runs only
* `axis experiments run <config>` must fail if the experiment already exists under baseline behavior
* repeated `resume` must not duplicate completed work

This package must expose those semantics faithfully, not reinterpret them for convenience.

---

### **10. End-to-End Integration Tests**

WP16 must include the first full user-path validation of the experimentation system.

The architecture’s testing strategy explicitly requires:

* CLI integration
* end-to-end workflows for at least:

  * one full single-run experiment
  * one full OFAT experiment
* resume validation
* artifact persistence and reconstruction validation
* deterministic behavior checks where appropriate

This package must therefore add true end-to-end tests that exercise:

* config file → CLI → executor → repository → persisted artifacts → CLI inspection

This is important because, up to now, the framework has been validated mostly through code-level integration. WP16 is the first point where actual user invocation paths must be proven.

---

## **Out of Scope**

Do **not** implement any of the following in WP16:

* business logic inside the CLI
* analysis subcommands
* custom metric computation
* visualization commands
* dashboards
* interactive prompts
* shell autocompletion systems
* rich TUI output
* filtering/query DSLs
* export pipelines beyond minimal JSON output
* database-backed repository support
* parallel execution options
* changes to executor semantics
* changes to repository semantics

Do not let the CLI become a second application layer.

---

## **Architectural Constraints**

The implementation must follow these rules.

### **1. Direct conceptual mapping**

The CLI must map directly to domain entities:

* `experiments`
* `runs`

and domain actions:

* `list`
* `run`
* `resume`
* `show`

No extra abstraction layer should be introduced. 

---

### **2. No business logic in CLI**

The CLI is not allowed to embed orchestration or resume logic.
It must call existing executors and repository methods.

---

### **3. Operate on persisted artifacts**

Inspection commands must operate from repository state, not from prior in-memory execution context.

---

### **4. Deterministic behavior**

Command behavior must be predictable and repeatable.
This matters especially for:

* resume
* show
* list
* error handling
* JSON output

The architecture explicitly frames deterministic and idempotent CLI behavior as a requirement.

---

### **5. Minimality**

The CLI must remain small.
The architecture explicitly warns against embedding analysis logic, visualization logic, or ad hoc command growth. 

---

## **Expected File Structure**

A reasonable extension could be:

```text
src/axis_system_a/
    run.py
    experiment.py
    repository.py
    experiment_executor.py
    cli.py
    __init__.py
```

If you prefer a package-style layout:

```text
src/axis_system_a/
    cli/
        __init__.py
        main.py
        commands.py
        formatting.py
```

That is acceptable only if it stays minimal.

The important thing is:

* argument parsing is separate from execution logic
* formatting is separate from repository/executor logic
* no excessive fragmentation

## **Testing Requirements**

Also create pytest tests for WP16.

The architecture is very explicit here, so the test surface should reflect that.

### **A. CLI command tests**

Validate at minimum:

* `axis experiments list`
* `axis experiments run`
* `axis experiments resume`
* `axis experiments show`
* `axis runs list`
* `axis runs show` 

### **B. Command routing tests**

Verify:

* correct command dispatch
* correct use of executors
* correct use of repository inspection methods
* explicit error handling for invalid argument combinations

These should remain logic tests, not formatting obsession tests.

### **C. End-to-end single-run workflow**

Test a full workflow:

1. write a small valid experiment config file
2. run `axis experiments run <config>`
3. verify persisted artifacts exist
4. run `axis experiments show <id>`
5. run `axis runs list --experiment <id>`
6. run `axis runs show <run_id>`
7. verify outputs are consistent with stored artifacts

This is one of the architecture’s required end-to-end cases. 

### **D. End-to-end OFAT workflow**

Test a full OFAT workflow:

1. write a small OFAT config file
2. execute through CLI
3. verify all resolved runs are created
4. verify summary reflects all runs in deterministic order
5. verify show/list commands expose the expected persisted state

This is also explicitly required. 

### **E. Resume workflow tests**

Validate via CLI:

* interrupted or partial experiment can be resumed
* only incomplete runs are re-executed
* completed runs are preserved
* repeated resume on completed experiment is a safe no-op

These must align with the existing resume semantics, not bypass them.

### **F. Error behavior tests**

Verify explicit and predictable failure behavior for cases such as:

* unknown experiment ID
* unknown run ID
* missing config file
* malformed config file
* invalid config content
* missing required artifacts during inspection
* invalid repository root

The architecture explicitly calls for clear failure behavior and explicit messages.

### **G. Determinism and idempotency tests**

Validate:

* repeated `resume` on completed experiment does not change state
* same config and seeds produce the same experiment-level core outputs
* CLI inspection reflects persisted truth consistently across repeated invocations

### **H. Test data strategy**

Use:

* small handcrafted configs
* small episode counts
* deterministic seeds
* temporary repository directories

Avoid:

* large sweeps
* unnecessary runtime complexity
* dependency on external infrastructure 

### **I. CLI test philosophy**

Do not turn CLI tests into fragile formatting tests.

The architecture explicitly says the priority is:

* correct command routing
* correct use of executors
* correct interaction with persisted artifacts
* explicit and predictable failure behavior 

Formatting details should only be tested when they materially affect machine use or user comprehension.

---

## **Implementation Style**

* Python 3.11
* explicit argument parsing
* no hidden magic
* thin command handlers
* deterministic error messages where practical
* use existing repository and executor layers
* keep JSON output explicit and stable

Avoid heavy CLI frameworks unless there is already a consistent project standard. A small standard-library-based CLI is perfectly fine here.

---

## **Expected Deliverable**

1. CLI entry point
2. baseline commands for experiments and runs
3. config-file-driven experiment execution
4. resume command wired to existing resume logic
5. inspection commands backed by persisted artifacts
6. minimal structured output support
7. end-to-end tests for single-run, OFAT, resume, inspection, and failure cases

---

## **Important Final Constraint**

WP16 must feel like:

> a **thin operational shell over the experimentation framework**
